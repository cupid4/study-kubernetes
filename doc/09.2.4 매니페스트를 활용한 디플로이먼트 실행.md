# 9.2.4 매니페스트를 활용한 디플로이먼트 실행

## 1. 실습 디렉터리 생성 및 이동

매니페스트(YAML) 파일을 깔끔하게 관리하기 위해 새로운 실습용 디렉터리를 만들고 이동하는 단계입니다.

```bash
eevee@myserver01:~$ cd work/                          # ❶ work 디렉터리로 이동
eevee@myserver01:~/work$ cd ch09/                     # ❸ 9장 실습 디렉터리로 이동
eevee@myserver01:~/work/ch09$ mkdir ex02               # ❺ 이번 실습을 위한 ex02 디렉터리 생성
eevee@myserver01:~/work/ch09$ cd ex02/                # ❼ 생성한 ex02 디렉터리로 이동

```

## 2. 디플로이먼트 매니페스트 파일 작성 (`deploy-test01.yml`)

텍스트 편집기(`vim`)를 열어 디플로이먼트를 정의하는 YAML 설정을 작성합니다.

```yaml
apiVersion: apps/v1                         # ❷ 쿠버네티스 API의 그룹과 버전을 지정합니다. (디플로이먼트는 apps/v1 사용)
kind: Deployment                            # ❸ 생성할 오브젝트 종류가 디플로이먼트(Deployment)임을 명시합니다.
metadata:                                   # ❹ 오브젝트의 메타데이터(이름 등) 설정 구문입니다.
  name: deploy-test01                       # ❺ 이 디플로이먼트의 이름을 'deploy-test01'로 지정합니다.
spec:                                       # ❻ 디플로이먼트의 명세(원하는 상태)를 정의합니다.
  replicas: 3                               # ❼ 생성할 파드(Pod)의 개수를 3개로 지정하여 레플리카셋을 제어합니다.
  selector:                                 # ❽ 디플로이먼트가 어떤 파드를 찾아서 관리할지 결정하는 스캐너입니다. 파드에 라벨을 붙이는 옵션으로 selector를 통해 디플로이먼트가 관리할 파드를 연결한다.
    matchLabels:                            # ❾ 일치해야 하는 라벨 조건을 정의합니다.
      app.kubernetes.io/name: web-deploy    # ❿ 이 라벨이 붙은 파드들을 관리 대상으로 삼습니다.
  template:                                 # ⑪ 실제 생성될 파드의 모양을 정의하는 템플릿(붕어빵 틀)입니다.
    metadata:                               # ⑫ 생성될 파드의 메타데이터입니다.
      labels:                               # ⑬ 파드에 부여할 라벨(이름표)을 설정합니다.
        app.kubernetes.io/name: web-deploy  # ⑭ selector.matchLabels(❿)와 반드시 일치해야 합니다.
    spec:                                   # ⑮ 파드 내부 사양(컨테이너)을 정의하는 구문입니다.
      containers:                           # ⑯ 파드 안에서 실행될 컨테이너 목록을 지정합니다.
        - name: nginx                       # ⑰ 컨테이너의 이름을 'nginx'로 지정합니다.
          image: nginx:latest               # ⑱ 컨테이너를 생성할 때 사용할 도커 이미지와 태그를 지정합니다.
```

### ⚠️ 중요 포인트 (셀렉터와 라벨의 일치)

* `spec.selector.matchLabels`에 적은 라벨(`❿`)과 `spec.template.metadata.labels`에 적은 라벨(``)은 **반드시 완벽하게 일치**해야 합니다.
* 만약 이 두 값이 다르면 쿠버네티스는 "내가 만들 파드 이름표와 내가 감시할 이름표가 다르다"며 에러를 발생시키고 배포를 거부합니다.


공유해 주신 281페이지 내용을 바탕으로 매니페스트를 활용한 디플로이먼트 생성 및 실행 결과를 마저 정리해 드립니다.

이번 페이지에서는 앞서 작성한 YAML 파일의 전체적인 구조를 시각적으로 확인하고, 이를 클러스터에 실제로 적용(`apply`)하여 파드들이 생성되는 과정을 다루고 있습니다.

## 3. [그림 9-3] 디플로이먼트 YAML 구조 요약

작성한 `deploy-test01.yml` 파일은 크게 두 영역으로 나뉩니다.

* **디플로이먼트 자체 설정:** API 버전, 종류, 이름, 유지할 파드 개수(`replicas: 3`), 그리고 관리할 파드를 콕 집어내는 셀렉터 이름표(`app.kubernetes.io/name: web-deploy`)가 정의되어 있습니다.
* **파드 템플릿 (점선 구역):** 디플로이먼트가 앞으로 찍어낼 파드의 붕어빵 틀입니다. 여기에 적힌 라벨 이름표가 위의 셀렉터와 일치해야 하며, 내부에 `nginx:latest` 이미지가 명시되어 있습니다.

## 4. 매니페스트 적용 및 리소스 확인

### 단계 1: 매니페스트 파일 배포 (`kubectl apply -f`)

`create` 대신 쿠버네티스에서 가장 권장되는 리소스 적용 명령어인 `apply`를 사용하여 YAML 파일을 클러스터에 반영합니다.

```bash
eevee@myserver01:~/work/ch09/ex02$ kubectl apply -f deploy-test01.yml
# 출력: deployment.apps/deploy-test01 created

```

### 단계 2: 연쇄 생성된 리소스 전체 확인 (`kubectl get deploy,rs,po`)

배포 후 약 1분이 지난 뒤 상태를 조회하면 디플로이먼트가 하위 리소스들을 줄줄이 감안하여 생성한 것을 볼 수 있습니다.

```bash
eevee@myserver01:~/work/ch09/ex02$ kubectl get deploy,rs,po

```

* **지휘관 (Deployment):** `deploy-test01`이 정상적으로 가동 중이며 3개의 파드가 모두 준비 완료되었습니다 (`3/3 READY`).
* **중간 관리자 (ReplicaSet):** 디플로이먼트 이름 뒤에 고유 해시값이 붙은 `deploy-test01-5545987bd6` 레플리카셋이 생성되어 목표 수량 3개를 유지하고 있습니다.
* **실제 일하는 방 (Pod):** 레플리카셋의 지휘 아래 `Running` 상태로 안정적으로 돌고 있는 **3개의 Nginx 파드 형제들**(`7qnnt`, `bq2m2`, `nb6ph`)을 확인할 수 있습니다.

### 💡 요약

명령어(CLI)로 디플로이먼트를 만들 때와 마찬가지로, YAML 파일(매니페스트)을 이용해 배포해도 [디플로이먼트 $\rightarrow$ 레플리카셋 $\rightarrow$ 파드 3개]로 이어지는 쿠버네티스의 3층 관리 구조가 완벽하게 생성됨을 로그로 확인할 수 있습니다.


공유해 주신 브라우저 화면(282페이지)을 바탕으로 **'9.2.4 매니페스트를 활용한 디플로이먼트 실행'의 마무리(삭제) 단계**와 새롭게 시작되는 '9.2.5 스케일(Scale)'의 준비 과정을 마크다운 형식으로 깔끔하게 정리해 드립니다.

## 5. 디플로이먼트 매니페스트 종료 (삭제)

실습이 완료된 리소스를 매니페스트 파일을 이용해 클러스터에서 깔끔하게 제거하는 단계입니다. 생성할 때와 마찬가지로 파일 지정 옵션(`-f`)을 사용합니다.

* **삭제 명령어:**
```bash
eevee@myserver01:~/work/ch09/ex02$ kubectl delete -f deploy-test01.yml
# 출력: deployment.apps "deploy-test01" deleted

```


* **삭제 확인:**
```bash
eevee@myserver01:~/work/ch09/ex02$ kubectl get deploy,rs,po
# 출력: No resources found in default namespace.

```


> `delete -f` 명령어를 사용하면 해당 YAML 파일로 생성되었던 디플로이먼트뿐만 아니라 하위의 레플리카셋, 파드까지 연쇄적으로 안전하게 일괄 삭제됩니다.

2. 파드의 수량을 조절하는 **스케일(Scale)** 실습을 위해 `ex02` 디렉터리를 복사하여 `ex03` 환경을 구축했습니다.
