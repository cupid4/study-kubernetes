# 9.5 스테이트풀셋

## 9.5.1 스테이트풀셋의 개념
### StatefulSet(스테이트풀셋)이란?

쿠버네티스에서 StatefulSet(스테이트풀셋)은 "상태가 있는(Stateful) 애플리케이션들"을 관리하기 위한 특별한 자원입니다.

일반적인 파드(Pod)들은 무작위로 생성되고 주소가 매번 바뀌며, 삭제되면 기존 데이터가 사라지는 '일회성' 특징을 가집니다. 반면, **스테이트풀셋으로 생성된 파드들은 다음과 같은 특별한 규칙**을 가집니다.

* **고유하고 고정된 이름 (식별자):** 파드가 순서대로 생성되며, `sfs-test01-0`, `sfs-test01-1` 처럼 뒤에 고정된 숫자가 붙습니다. 파드가 죽었다가 다시 켜져도 이 이름은 절대 바뀌지 않습니다.
* **고정된 저장소 (데이터 보존):** 각각의 파드는 자신만의 전용 저장소(Volume)를 가집니다. 파드가 재시작되더라도 원래 자기가 쓰던 저장소와 자동으로 다시 연결되어 데이터가 유지됩니다.
* **순서 있는 생성과 삭제:** 0번 파드가 완전히 켜져야 1번 파드가 켜지고, 꺼질 때도 역순으로 안전하게 꺼집니다.

> 💡 **주로 언제 쓰나요?**
> 데이터의 연속성과 순서가 매우 중요한 **데이터베이스(MySQL, MongoDB, PostgreSQL)**나 **분산 시스템(Kafka, Elasticsearch)** 등을 쿠버네티스에서 안정적으로 돌릴 때 필수적으로 사용됩니다.

---

#### 주석이 포함된 StatefulSet YAML 파일

설명해 드린 개념을 바탕으로, 작성하셨던 YAML 파일에 상세한 주석을 결합한 최종 버전입니다.

```yaml
apiVersion: apps/v1                      # ② 버전을 지정합니다 (StatefulSet을 쓰기 위한 기본 규격입니다).
kind: StatefulSet                        # ③ 생성할 오브젝트 종류를 순서가 보장되는 '스테이트풀셋'으로 지정합니다.
metadata:                                # ④ 이 오브젝트의 이름표를 달아주는 기본 정보 칸입니다.
  name: sfs-test01                       # ⑤ 이 스테이트풀셋의 이름을 'sfs-test01'로 정합니다.
spec:                                    # ⑥ [상세 설계] 이 스테이트풀셋이 어떤 상태를 유지해야 하는지 상세 설정을 시작합니다.
  replicas: 3                            # ⑦ [파드 개수] 실행할 파드의 개수를 3개로 지정합니다. (하나가 죽으면 자동으로 다시 켜집니다.)
  selector:                              # ⑧ ⑨ ⑩ [관리 대상 지정] 'web-sfs01'이라는 명찰(Label)을 달고 있는 파드들만 내가 관리하겠다고 선언합니다.
    matchLabels:      
      app.kubernetes.io/name: web-sfs01       
  serviceName: sfs-service01             # ⑪ [연결할 서비스] 파드들에게 고정된 도메인(이름) 주소를 부여해줄 헤드리스 서비스의 이름을 적어줍니다.
  template:                              # ⑫ [붕어빵 틀] 새로운 파드들을 똑같이 찍어내기 위한 구체적인 설계도(붕어빵 틀)를 시작합니다.
    metadata:                            # ⑬ ⑭ ⑮ [파드 명찰 지정] 여기서 태어날 파드들에게 'web-sfs01' 명찰을 붙여줍니다. (위의 selector가 이 명찰을 보고 관리합니다.)
      labels:     
        app.kubernetes.io/name: web-sfs01   
    spec:                                # ⑯ [파드 내부 설계] 파드가 생성될 때 내부에 어떤 프로그램을 실행할지 설정을 시작합니다.
      containers:                        # ⑰ [컨테이너 설정] 파드 안에서 실질적으로 돌아갈 프로그램(컨테이너)들의 목록입니다.
      - name: nginx                      # ⑱ 실행할 프로그램(컨테이너)의 이름을 'nginx'로 정합니다.
        image: nginx:latest              # ⑲ 프로그램의 원본 파일인 이미지(nginx의 가장 최신 버전)를 다운로드받아 실행하겠다고 정합니다.

```

## 9.5.2 헤드리스 서비스
 - 개별 네트워크를 식별하기 위해 필요한 서비스로 대문 IP(ClusterIP)가 없는, 이름 뿐인 서비스이다.
 - 일반적인 쿠버네티스 서비스는 파드들을 대표하는 단 하나의 가상 IP(ClusterIP)를 가집니다. 사용자가 그 대표 IP로 접속하면 서비스가 알아서 여러 파드 중 하나로 트래픽을 토스해주죠.
 - 하지만 헤드리스 서비스는 clusterIP: None 설정을 통해 이 대표 IP를 아예 없애버립니다.


헤드리스 서비스(Headless Service)는 한 마디로 "대문 IP(ClusterIP)가 없는, 이름뿐인 서비스"입니다.

일반적인 쿠버네티스 서비스는 파드들을 대표하는 단 하나의 가상 IP(ClusterIP)를 가집니다. 사용자가 그 대표 IP로 접속하면 서비스가 알아서 여러 파드 중 하나로 트래픽을 토스해주죠.

하지만 헤드리스 서비스는 **`clusterIP: None`** 설정을 통해 이 대표 IP를 아예 없애버립니다.

### 9.5.2.1. 헤드리스 서비스는 왜 쓸까요? (핵심 목적)

> **"쿠버네티스 대표 IP 뒤에 숨지 말고, 파드들의 진짜 IP 주소를 다 공개해!"**

일반 서비스는 파드들의 IP를 꽁꽁 숨기고 대표 IP만 보여주지만, 헤드리스 서비스는 이 서비스를 바라보는 **모든 파드들의 개별 IP 주소를 외부에 그대로 노출**시킵니다.

이를 통해 클라이언트가 특정 파드와 **1:1로 직접 통신**할 수 있게 만듭니다.


### 9.5.2.2. 일반 서비스 vs 헤드리스 서비스 차이점

#### 일반 서비스 (ClusterIP가 있는 경우)

* **요청:** 사용자가 `my-service`라는 도메인(이름)을 호출합니다.
* **DNS 응답:** 쿠버네티스 DNS가 서비스의 **대표 가상 IP(예: `10.96.0.1`)** 딱 하나만 알려줍니다.
* **동작:** 무조건 대표 IP를 거쳐서 랜덤하게 파드로 연결됩니다.

#### 헤드리스 서비스 (`clusterIP: None`)

* **요청:** 사용자가 헤드리스 서비스의 이름(`my-headless-service`)을 호출합니다.
* **DNS 응답:** 쿠버네티스 DNS가 대표 IP 대신, 이 서비스에 묶여있는 모든 파드들의 개별 실제 IP 목록(예: `192.168.1.11`, `192.168.1.12`)을 전부 다 반환합니다.
* **동작:** 사용자가 파드들의 실제 IP 주소를 다 알게 되므로, 원하는 특정 파드를 골라서 직접 연결할 수 있습니다.

### 9.5.2.3. 주로 어떤 경우에 쓰나요?

바로 다음 단원에 나올 스테이트풀셋(StatefulSet)이나 **데이터베이스(DB) 클러스터**를 구축할 때 필수적으로 사용됩니다.

* **DB 마스터-슬레이브 구조:** 쓰기 작업은 '마스터 파드'에만 보내고, 읽기 작업은 '슬레이브 파드'에만 보내야 할 때, 파드 각각의 진짜 주소를 알아야 하므로 헤드리스 서비스를 씁니다.
* **파드끼리 서로 통신해야 할 때:** 카프카(Kafka), 엘라스틱서치(Elasticsearch) 같은 분산 시스템은 파드들끼리 서로 동기화를 위해 직접 통신해야 합니다. 이때 서로의 실제 주소를 알아내는 창구로 사용됩니다.

### 💡 요약하자면

일반 서비스가 트래픽을 골고루 나눠주는 **'로드밸런서/안내원'** 역할이라면, 헤드리스 서비스는 트래픽 분산은 하지 않고 뒤에 있는 파드들의 주소록만 전달해주는 **'전화번호부'** 역할을 지닌 서비스입니다.

## 9.5.3 스테이트풀셋 생성

![alt text](<asset/09.5 스테이트풀셋/image.png>)


스테이트풀셋(StatefulSet)과 헤드리스 서비스(Headless Service)가 서로 어떻게 이름을 매칭하여 연결되는지 그 관계를 시각적으로 보여주는 다이어그램입니다.

쉽게 말해, "얘네 둘이 서로를 어떻게 알아보고 한 팀으로 묶이는가?"에 대한 설명서라고 보시면 됩니다.

핵심적인 연결 고리는 크게 두 가지 흐름으로 나뉩니다.

---

### 9.5.3.1. 서비스가 스테이트풀셋(파드)을 찾아가는 방법 (초록색 상자)

왼쪽의 서비스와 오른쪽의 스테이트풀셋은 `app.kubernetes.io/name: web-sfs01`이라는 라벨(Label)을 통해 한 팀이 됩니다.

* **서비스의 Selector (왼쪽):** "나는 라벨 이름이 `web-sfs01`인 파드들을 관리할 거야!"라고 타겟을 지정합니다 (`selector.app.kubernetes.io/name`).
* **스테이트풀셋의 Selector (오른쪽 위):** "내가 앞으로 관리할 라벨 기준은 `web-sfs01`이야!"라고 선언합니다 (`selector.matchLabels`).
* **스테이트풀셋의 Template (오른쪽 아래):** "그래서 내가 생성하는 모든 파드들에는 `web-sfs01`이라는 명찰(라벨)을 붙여서 태어 나게 할게!"라고 설정합니다 (`template.metadata.labels`).

> 💡 **결과:** 서비스의 selector와 스테이트풀셋이 생성한 파드의 라벨명이 **`web-sfs01`**로 정확히 일치하기 때문에, 서비스는 이 스테이트풀셋이 생성한 파드들을 올바르게 찾아가서 연결할 수 있습니다.

---

### 9.5.3.2. 스테이트풀셋이 서비스를 지정하는 방법 (보라색 상자)

스테이트풀셋으로 만들어진 파드들은 각각 고유하고 고정된 네트워크 식별자(도메인 이름)를 가집니다. 이를 가능하게 하려면 스테이트풀셋 자체가 본인과 연결될 서비스의 이름을 명시해 주어야 합니다.

* **서비스의 이름 (왼쪽):** 헤드리스 서비스의 이름이 `sfs-service01`로 정의되어 있습니다 (`metadata.name`).
* **스테이트풀셋의 서비스 지정 (오른쪽):** 스테이트풀셋 내부 설정에서 "내 네트워크를 담당해 줄 서비스는 `sfs-service01`이야!"라고 서비스 이름을 직접 적어 줍니다 (`spec.serviceName`).

> 💡 **결과:** 스테이트풀셋에 **`serviceName: sfs-service01`**을 명시함으로써, 나중에 생성될 파드들이 `파드이름.sfs-service01`과 같은 고정된 도메인 주소(FQDN)를 가지고 서로 통신할 수 있게 됩니다.


### 9.5.3.3. 한눈에 요약하자면?

* **보라색 연결선 (`serviceName`):** 스테이트풀셋이 서비스를 향해 **"너가 내 네트워크 이름표를 제공해 줘"** 하고 요청하는 연결입니다.
* **초록색 연결선 (`Labels`):** 서비스가 파드들을 향해 **"너희 명찰(`web-sfs01`)을 보니 내가 관리할 녀석들이 맞구나!"** 하고 묶어주는 연결입니다.

이 두 가지 연결이 모두 정확하게 맞아떨어져야만 쿠버네티스에서 데이터의 연속성이 보장되는 안정적인 스테이트풀셋 환경이 완성됩니다.

## 9.5.4 스테이트풀셋과 헤드리스 서비스와의 관계

스테이트풀셋(StatefulSet)과 헤드리스 서비스(Headless Service)는 쿠버네티스에서 늘 세트로 붙어 다니는 **실과 바늘 같은 관계**입니다.

결론부터 말하면, **"스테이트풀셋이 만든 파드들에게 고정된 개별 주소(도메인)를 만들어주기 위해 헤드리스 서비스가 필요"** 합니다.

이 둘의 관계를 이해하기 쉽게 세 가지 포인트로 정리해 드릴게요.


### 9.5.4.1. 일반 서비스와 헤드리스 서비스의 차이

* **일반 서비스:** 대표 전화번호(하나의 IP)를 두고, 전화가 오면 여러 파드 중 한 곳으로 랜덤하게 연결해 줍니다. (누가 받아도 상관없는 구조)
* **헤드리스 서비스 (`clusterIP: None`):** 대표 번호가 없습니다. 대신 전화번호부 역할을 하여, 각각의 파드로 바로 연결되는 개별 직통 번호(고유 도메인)를 모두 공개합니다.

---

### 9.5.4.2. 왜 스테이트풀셋은 헤드리스 서비스가 필요할까?

데이터베이스 같은 애플리케이션은 파드들이 서로를 명확히 구별할 수 있어야 합니다. 예를 들어 "0번 파드는 마스터(쓰기 전용), 1번과 2번 파드는 슬레이브(읽기 전용)" 같은 규칙이 있다면, **1번 파드는 랜덤이 아니라 반드시 0번 파드를 정확히 찾아가서 데이터를 복사**해와야 합니다.

이때 헤드리스 서비스가 각 파드에 다음과 같이 예측 가능하고 고정된 이름(도메인 주소)을 만들어 줍니다.

$$\text{파드 이름} . \text{헤드리스 서비스 이름} . \text{네임스페이스} . \text{svc.cluster.local}$$

위에서 작성한 YAML 파일을 예로 들면 파드들은 각각 아래와 같은 고유한 주소를 가지게 됩니다.

* `sfs-test01-0.sfs-service01.default.svc.cluster.local`
* `sfs-test01-1.sfs-service01.default.svc.cluster.local`
* `sfs-test01-2.sfs-service01.default.svc.cluster.local`

파드가 죽었다가 다시 살아나도 이 주소는 절대 바뀌지 않기 때문에, 파드들끼리 서로를 잃어버리지 않고 안정적으로 통신할 수 있습니다.

---

### 9.5.4.3. 핵심 요약

* **스테이트풀셋**은 파드들에게 **고정된 이름(`-0`, `-1`, `-2`)과 영구적인 저장소**를 쥐어줍니다.
* **헤드리스 서비스**는 그 고정된 이름들을 네트워크 주소로 변환하여 서로 직통으로 찾아갈 수 있는 길(고정 DNS)을 열어줍니다.

> 💡 **그래서 규칙입니다!**
> 스테이트풀셋을 만들 때는 항상 `spec.serviceName`에 매칭할 헤드리스 서비스의 이름을 명시해 주어야 비로소 완벽한 스테이트풀셋 환경이 완성됩니다.
