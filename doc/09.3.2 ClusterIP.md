# 9.3.2 ClusterIP

## 1. ClusterIP 서비스의 정의 및 특징

* **기본 설정값(Default):** 쿠버네티스에서 서비스를 생성할 때 별도의 타입을 지정하지 않으면 가장 기본적으로 설정되는 유형입니다.
* **클러스터 내부 전용:** 쿠버네티스 클러스터 내부에서만 접근할 수 있는 고정된 가상 IP를 할당합니다.
* **외부 접근 차단:** 클러스터 외부에 있는 사용자나 외부 시스템은 이 IP를 통해 파드에 접근할 수 없습니다. 즉, **클러스터 내부의 파드들끼리 서로 통신할 때만 사용**되는 폐쇄적인 서비스입니다.

## 쿠버네티스 ClusterIP 서비스 실습 및 요약 (p.293 ~ 299)

### 1. ClusterIP 서비스 개요 (p.293 ~ 294)

* **기본 서비스 타입:** 쿠버네티스에서 `spec.type`을 별도로 지정하지 않으면 자동으로 설정되는 기본 네트워크 서비스 유형입니다.
* **클러스터 내부 전용 통신:** 클러스터 내부에서만 유효한 고유 IP(ClusterIP)를 할당받으며, 외부 네트워크에서는 이 IP로 접근할 수 없습니다. 클러스터 내의 파드(Pod)나 컴포넌트 간 안전한 내장 로드밸런싱과 라우팅을 목적으로 사용됩니다.

### 2. ClusterIP 생성 및 확인 (p.295 ~ 296)

* **서비스 생성:** 준비된 YAML 파일(`service-test01.yml`)을 이용해 `web-service`라는 이름의 ClusterIP 서비스를 생성합니다.
* **라우팅 연동:** 서비스가 생성되면 해당 서비스의 셀렉터(Selector) 조건과 일치하는 파드들(디플로이먼트/레플리카셋으로 관리되는 파드 집합)이 자동으로 서비스 뒤에 연결됩니다.
* **상태 확인:** `kubectl get service` 명령어로 생성된 서비스의 내부 IP와 포트 번호(예: `10.107.43.35:80`)를 확인합니다.

![alt text](<asset/09.3.2 ClusterIP/image-1.png>)

### 3. 내부 접근 테스트 실습 (p.297 ~ 298)

* **테스트용 파드 생성:** 클러스터 내부 통신을 검증하기 위해 별도의 Nginx 테스트 파드(`nginx01`)를 기동합니다.
* **파드 내부 접속:** `kubectl exec -it nginx01 -- /bin/bash` 명령어를 사용해 실행 중인 `nginx01` 컨테이너의 셸(Shell) 환경으로 접속합니다.
* **통신 검증:** 컨테이너 내부에서 앞서 확인한 서비스의 내부 IP로 `curl 10.107.43.35` 요청을 보냅니다. 서비스가 요청을 받아 연결된 파드들로 트래픽을 올바르게 전달하고 정상적인 응답을 반환하는지 최종 확인한 후 `exit`로 빠져나옵니다.

![alt text](<asset/09.3.2 ClusterIP/image-2.png>)



### 4. 실습 자원 정리 (p.299)

실습이 모두 정상적으로 완료되면 클러스터 자원을 낭비하지 않도록 아래 명령어를 통해 생성했던 서비스, 디플로이먼트, 테스트 파드를 역순으로 깔끔하게 삭제합니다.

```bash
# 1. 쿠버네티스 서비스 종료
kubectl delete -f service-test01.yml

# 2. 디플로이먼트 종료
kubectl delete -f deploy-test01.yml

# 3. 테스트용 Nginx 파드 종료
kubectl delete -f nginx-test01.yml

# 4. 남은 파드가 없는지 최종 확인
kubectl get pod
# (출력: No resources found in default namespace.)

```
