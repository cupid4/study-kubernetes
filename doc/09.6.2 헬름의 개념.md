# 9.6.2 헬름의 개념
 - yaml 파일을 만들지 않고, 쿠버네티스 환경에서 애플리케이션을 쉽고 편하게 설치할 수 있다.

![alt text](<asset/09.6.2 헬름의 개념/image.png>)


공유해주신 370페이지의 [그림 9-34 헬름의 개념]은 쿠버네티스의 패키지 매니저인 Helm(헬름)이 어떻게 작동하고 구성되는지 한눈에 보여주는 아키텍처 지도입니다.

쉽게 말해, 스마트폰에서 앱을 설치할 때 '앱스토어'에서 다운받아 클릭 한 번으로 설치하듯, 쿠버네티스에서도 복잡한 여러 개의 YAML 파일들을 하나로 묶어 쉽게 설치하고 관리할 수 있게 도와주는 **'쿠버네티스용 앱스토어'** 시스템입니다.

그림에 나오는 3가지 핵심 영역을 기준으로 역할을 설명해 드릴게요.

---

## 1. 🌐 헬름 차트 리포지토리 (Helm Chart Repository)

* **역할:** 전 세계 개발자들이 미리 만들어 놓은 쿠버네티스 실행 패키지(예: Nginx, MySQL, WordPress 등)들이 모여 있는 공용 저장소(앱스토어)입니다.
* **동작:** 도커 허브와 비슷하게, 내가 만든 패키지를 저장소에 올릴 수도 있고(`push`), 필요한 패키지를 내 환경으로 다운로드(`pull`) 받아올 수도 있습니다.

## 2. 🧑‍💻 관리자 & 헬름 클라이언트 (Helm Chart)

관리자(사용자님)가 내 PC나 터미널에서 명령어를 내리는 구역입니다. 이곳에서 다운로드한 '헬름 차트'를 열어보게 되는데, 내부 구조가 크게 두 가지로 나뉩니다.

* **템플릿 (Template):** 기본 구동에 필요한 복잡하고 고정된 쿠버네티스 YAML(Service, Deployment 등)들의 뼈대 소스코드입니다.
* **values.yaml (설정 파일):** **이 그림에서 가장 중요한 파일입니다.** 복잡한 템플릿 코드를 일일이 수정할 필요 없이, 이 파일 안에 있는 몇 가지 변수(예: '서버 개수는 3개', '포트 번호는 80')만 내 입맛대로 수정하면 나만의 맞춤형 패키지가 완성됩니다.

## 3. ☸️ 쿠버네티스 클러스터 (마스터 노드)

* **역할:** 최종적으로 내 애플리케이션이 실행되는 실제 서버 환경입니다.
* **동작:** 헬름 클라이언트에서 `헬름 설치(helm install)` 명령을 내리면, 커스텀된 헬름 차트가 쿠버네티스의 '마스터 노드'로 전달됩니다. 책의 본문 설명처럼 워커 노드마다 일일이 설치할 필요 없이, 대문 역할인 마스터 노드가 명령을 받아 전체 클러스터에 앱을 배포하게 됩니다.

---

## 💡 그림으로 보는 전체 흐름 요약

1. **Pull:** 리포지토리(앱스토어)에서 원하는 프로그램의 '헬름 차트'를 다운로드한다.
2. **Modify:** `values.yaml` 파일을 수정해 내 환경에 맞게 옵션(포트, 용량 등)을 커스텀한다.
3. **Install:** 마스터 노드에 설치 명령을 내려 쿠버네티스 클러스터에 한 번에 배포한다.

## Helm 정의

- Helm(헬름)을 한 마디로 정의하자면 '쿠버네티스 전용 패키지 매니저(앱스토어)'입니다.
- 스마트폰에 앱스토어가 있고, Mac에서 터미널 유틸리티를 설치할 때 Homebrew를 사용하며, Spring Boot나 Kotlin 프로젝트에서 복잡한 라이브러리 의존성을 `build.gradle` 파일 하나로 우아하게 관리하듯, 쿠버네티스 생태계에는 헬름이 그 역할을 담당합니다.


### 1. 헬름이 없다면 겪게 되는 'YAML 지옥'

쿠버네티스에 그럴듯한 시스템 하나를 띄우려면 앞서 실습하셨던 것처럼 수많은 파일이 필요합니다.
만약 쿠버네티스에 'Redis 데이터베이스'나 '웹 서버'를 구축한다고 가정해 볼까요?

* 파드를 띄울 `Deployment.yaml`
* 외부와 연결할 `Service.yaml`
* 스토리지를 묶어줄 `PV.yaml`, `PVC.yaml`
* 환경 변수를 담을 `ConfigMap.yaml`
* 비밀번호를 담을 `Secret.yaml`

이 모든 파일을 직접 타이핑해서 만들고, `kubectl apply -f`를 6번씩 쳐야 합니다. 버전이 업데이트되거나 포트 번호 하나를 바꾸려면 저 수많은 파일들을 열어서 일일이 수정해야 하는 끔찍한 유지보수 지옥이 열립니다.

### 2. 헬름의 마법: "다 묶어놨으니 변수만 넣으세요!"

전 세계의 천재 개발자들이 생각했습니다. *"어차피 Redis나 Nginx 설치하는 뼈대 코드는 다 똑같은데, 이걸 하나의 패키지로 묶어버리자!"*

이렇게 복잡한 YAML 파일들을 하나로 예쁘게 압축해 놓은 묶음을 헬름에서는 '차트(Chart)'라고 부릅니다.
우리는 뼈대 코드를 건드릴 필요 없이, `values.yaml`이라는 단 하나의 설정 파일에 *"포트 번호는 8080으로 해주고, 스토리지는 10GB만 줘"* 라고 적은 뒤 아래 명령어 한 줄만 치면 끝납니다.

```bash
helm install my-redis bitnami/redis

```

마치 `brew install`이나 `apt-get`을 치는 것처럼 수십 개의 쿠버네티스 자원이 단 3초 만에 촥! 하고 설치됩니다.

---

### 3. 실무에서는 언제, 어떻게 사용하나요?

실제 현업에서는 크게 두 가지 상황에서 헬름을 숨 쉬듯 사용합니다.

* **검증된 외부 인프라/솔루션을 즉시 도입할 때 (가장 많이 씀)**
* 책에서 배우고 계신 **Nginx Ingress Controller**를 설치할 때.
* RAG(검색 증강 생성) 시스템 구성을 위해 **Qdrant** 같은 벡터 DB를 쿠버네티스 위에 안정적으로 배포해야 할 때.
* Prometheus, Grafana 같은 복잡한 시스템 모니터링 도구를 클릭 한 번(명령어 한 줄) 수준으로 설치할 때.


* **우리 회사의 서비스를 찍어내는 '템플릿'을 만들 때**
* 직접 개발한 API 서버나 파이프라인을 배포할 때마다 YAML을 복사&붙여넣기 하는 대신, 우리만의 '헬름 차트'를 만듭니다.
* 개발 환경(Dev)에 배포할 때는 `values-dev.yaml`(파드 1개), 운영 환경(Prod)에 배포할 때는 `values-prod.yaml`(파드 10개)만 갈아 끼우면 안전하고 빠르게 배포를 통제할 수 있습니다.
