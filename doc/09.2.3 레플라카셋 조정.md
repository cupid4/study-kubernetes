# 9.2.3 레플리카셋 조정

## 1. 레플리카셋 조정을 위한 기본 원칙

* 레플리카셋(ReplicaSet)은 사용자가 지정한 파드(Pod)의 개수를 그대로 유지하고 가용성을 보장하는 컨트롤러입니다.
* 쿠버네티스 운영 시 파드의 개수를 늘리거나 줄여야 할 때, **레플리카셋 오브젝트를 직접 수정하거나 다루지 않는 것이 원칙**입니다.
* 상위 관리자인 **디플로이먼트(Deployment)를 통해** 파드의 개수를 조절하면, 디플로이먼트가 백엔드에서 레플리카셋을 알아서 안전하게 조정합니다.

---

## 2. 명령어로 레플리카셋 개수 지정하여 배포하기

디플로이먼트를 처음 생성할 때 `--replicas` 옵션을 사용하면 원하는 파드의 개수를 명시하여 배포할 수 있습니다.

* **실행 명령어:**
```bash
kubectl create deployment deploy-nginx --image=nginx --replicas=3

```


* `--replicas=3`: 이 옵션을 통해 nginx 웹서버 파드를 처음부터 동시에 3개 구동하도록 지정합니다.
* 실행 결과로 `deployment.apps/deploy-nginx created` 메시지가 출력되며 디플로이먼트가 정상 생성됩니다.


### 💡 핵심 요약

1. 파드의 수량을 관리하는 것은 **레플리카셋**의 역할이다.
2. 하지만 실무에서는 레플리카셋을 직접 건드리지 않고, 항상 **디플로이먼트를 통해서 간접적으로 수량을 조절**한다.
3. 처음 디플로이먼트를 만들 때 **`--replicas=개수`** 옵션을 주면 원하는 수량만큼 파드를 동시에 띄울 수 있다.



현재 열고 계신 교재 276페이지의 **`kubectl get deploy,rs,po -o wide`** 명령어 실행 및 결과 분석 내용을 이어서 Markdown(MD) 형식으로 정리해 드립니다.

---

## 3. 상세 정보 조회를 통한 구조 분석 (`-o wide`)

`-o wide` 옵션을 추가하면 파드가 어느 서버(노드)에 배포되었는지, 어떤 IP를 할당받았는지 등 상세한 네트워크 및 컨테이너 정보를 한눈에 확인할 수 있습니다.

* **실행 명령어:**
```bash
kubectl get deploy,rs,po -o wide

```



### 🔍 출력 결과 분석

```markdown
# 1. 디플로이먼트 정보 (지휘관)
NAME                          READY   UP-TO-DATE   AVAILABLE   AGE     CONTAINERS   IMAGES   SELECTOR
deployment.apps/deploy-nginx  3/3     3            3           2m32s   nginx        nginx    app=deploy-nginx

```

* **CONTAINERS / IMAGES:** 이 디플로이먼트가 `nginx` 이미지를 사용해 컨테이너를 구동하고 있음을 보여줍니다.
* **SELECTOR (`app=deploy-nginx`):** 디플로이먼트가 하위 리소스들을 식별하고 관리하기 위한 핵심 라벨 조건입니다.

```markdown
# 2. 레플리카셋 정보 (중간 관리자)
NAME                                    DESIRED   CURRENT   READY   AGE     CONTAINERS   IMAGES   SELECTOR
replicaset.apps/deploy-nginx-7d74c85c6f   3         3         3       2m32s   nginx        nginx    app=deploy-nginx,pod-template-hash=7d74c85c6f

```

* 디플로이먼트의 명령을 받아 지정된 수량(**DESIRED: 3**)을 정확히 유지하고 있습니다.
* Selector에 `pod-template-hash`가 자동으로 추가되어 자신이 관리해야 할 파드들을 고유하게 지정합니다.

```markdown
# 3. 파드 정보 (실제 일하는 방)
NAME                                  READY   STATUS    RESTARTS   AGE     IP                NODE         ...
pod/deploy-nginx-7d74c85c6f-967d7     1/1     Running   0          2m32s   192.168.149.132   myserver03   ...

```

* **STATUS (`Running`):** 앞서 다룬 `hello-world`와 달리, `nginx`는 백그라운드에서 계속 실행되는 웹서버이기 때문에 세 개의 파드 모두 안정적으로 `Running` 상태를 유지합니다.
* **IP:** 쿠버네티스 내부(Calico CNI가 할당한)에서 가상 통신에 사용하는 파드 각각의 고유 IP 주소입니다.
* **NODE (`myserver03` 등):** 각 파드가 클러스터 내의 어떤 서버(노드) 위에 배치되어 자원을 쓰고 있는지 보여줍니다. 쿠버네티스의 스케줄러가 여러 노드에 파드를 알아서 골고루 분산 배치해 줍니다.

---

### 💡 핵심 요약

1. `--replicas=3` 옵션으로 배포된 **Nginx 웹서버 파드 3개는 죽지 않고 정상적으로 `Running` 상태를 유지**합니다.
2. `-o wide` 옵션을 사용하면 각 파드에 할당된 **가상 IP**와 해당 파드가 실제로 살아 숨 쉬고 있는 **물리/가상 서버(NODE)** 위치를 명확하게 추적할 수 있습니다.

## 4. 파드 삭제를 통한 자동 복구(Self-Healing) 검증

디플로이먼트가 관리하는 파드 중 하나를 사용자가 인위적으로 삭제했을 때, 쿠버네티스가 시스템을 어떻게 스스로 치유하고 복구하는지 확인하는 실습입니다.

* **단계 1: 특정 파드 강제 삭제 (`kubectl delete pod`)**
실행 중인 3개의 Nginx 파드 중 하나(`deploy-nginx-7d74c85c6f-qzvlx`)를 명시하여 삭제합니다.
```bash
eevee@myserver01:~$ kubectl delete pod deploy-nginx-7d74c85c6f-qzvlx
# 출력: pod "deploy-nginx-7d74c85c6f-qzvlx" deleted

```


* **단계 2: 클러스터 상태 재조회 (`kubectl get deploy,rs,po`)**
파드를 삭제한 직후 다시 상태를 조회하면, 파드가 하나 줄어드는 것이 아니라 여전히 **3/3 READY** 상태를 유지하고 있는 것을 볼 수 있습니다.
```bash
eevee@myserver01:~$ kubectl get deploy,rs,po
NAME                          READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/deploy-nginx  3/3     3            3           13m

```



### 🔍 무대 뒤에서 일어난 자동 복구 메커니즘 ([그림 9-2] 참고)

1. **센서 감지:** 사용자가 파드를 지우는 순간, 중간 관리자인 레플리카셋(ReplicaSet)**이 클러스터를 감시하다가 `"어? 셀렉터(`app=deploy-nginx`)로 조회되는 파드가 내가 유지해야 하는 목표치(**DESIRED: 3**)보다 하나 부족한 2개네?"* 하고 즉각 알아차립니다.
2. **즉시 재생성:** 레플리카셋은 지체 없이 동일한 설정을 가진 **새로운 파드를 즉시 하나 더 생성**하여 수량 3개를 강제로 맞추어 버립니다.
3. **결론:** 파드들은 여러 노드(`myserver02`, `myserver03`)에 흩어져서 배포되어 있지만, 결국 하나의 레플리카셋이라는 지휘 아래에 묶여 있기 때문에 **사용자가 파드를 아무리 지워도 디플로이먼트를 삭제하지 않는 한 지정된 개수만큼 끊임없이 되살아납니다.**
