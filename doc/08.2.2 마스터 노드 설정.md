# 8.2.2 마스터 노드 설정 및 클러스터 구축 개요

마스터 노드 설정 단계는 모든 서버(노드)에 쿠버네티스 프로그램 설치를 마친 후, 특정 서버(`myserver01`)를 클러스터 전체를 지휘하고 제어하는 **마스터 노드(Control Plane)**로 지정하는 과정입니다.

---

## 1. 인증서 유효기간 확인 (`kubeadm certs check-expiration`)
마스터 노드를 본격적으로 초기화하기 전, 내부 통신에 필수적인 디지털 인증서들의 유효 상태를 점검합니다.
* **참고 (`!MISSING!` 에러):** 초기화 전에는 인증서 파일이 생성되지 않은 상태이므로, 항목 옆에 빨간색으로 `!MISSING!` 문구가 출력되는 것이 정상입니다.

---

## 2. 이미지 사전 다운로드 및 CRI 중복 에러 해결
쿠버네티스 구동에 필요한 핵심 컴포넌트 이미지 목록을 확인하고 이를 서버로 미리 다운로드(Pull)합니다.

* **필요 이미지 목록 확인:**
  ```bash
  kubeadm config images list

```

* **CRI 중복 에러 해결 후 다운로드:**
호스트에 `containerd`와 `cri-dockerd` 엔진이 동시에 감지되면 `Found multiple CRI endpoints...` 에러가 발생하며 멈춥니다. 이를 해결하기 위해 아래와 같이 사용할 엔진 소켓을 명시해야 합니다.
```bash
sudo kubeadm config images pull --cri-socket /run/containerd/containerd.sock

```



---

## 3. 마스터 노드 초기화 실행 (`kubeadm init`)

아래 명령어를 실행하여 마스터 노드를 최종적으로 활성화합니다. 통신할 대표 IP 주소와 컨테이너용 가상 네트워크 대역을 지정합니다.

```bash
sudo kubeadm init --apiserver-advertise-address=10.0.2.4 --pod-network-cidr=192.168.0.0/16 --cri-socket /run/containerd/containerd.sock

```

* **성공 메시지:** 정상적으로 완료되면 `"Your Kubernetes control-plane has initialized successfully!"` 문구가 출력됩니다.
* **⚠️ 주의 (워커 노드 join 토큰 보관):** 출력 결과 맨 마지막에 생성되는 `kubeadm join [IP:포트] --token ... --discovery-token-ca-cert-hash ...` 형태의 두 줄짜리 명령어는 **워커 노드들을 클러스터에 연결할 때 사용하는 유일한 인증 토큰**이므로 반드시 메모장에 따로 복사해 두어야 합니다.

---

## 4. 일반 사용자 환경 설정 (루트 세션 종료)

보안 및 편의성을 위해 일반 사용자 계정(`eevee`)에서도 관리자 명령어(`kubectl`)를 내릴 수 있도록 권한을 설정합니다. `root` 세션을 종료한 후 다음 명령을 수행합니다.

```bash
exit  # root 계정 탈출
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config

```

---

## 5. 네트워크 플러그인(Calico CNI) 설치 및 확인

쿠버네티스는 초기화 직후 컨테이너 간 통신망이 없는 상태이므로, 가상 네트워크 인터페이스(CNI)인 **Calico**를 배포합니다.

* **Tigera Operator 및 커스텀 리소스 배포:**
```bash
kubectl create -f [https://raw.githubusercontent.com/projectcalico/calico/v3.26.3/manifests/tigera-operator.yaml](https://raw.githubusercontent.com/projectcalico/calico/v3.26.3/manifests/tigera-operator.yaml)
curl [https://raw.githubusercontent.com/projectcalico/calico/v3.26.3/manifests/custom-resources.yaml](https://raw.githubusercontent.com/projectcalico/calico/v3.26.3/manifests/custom-resources.yaml) -O
kubectl create -f custom-resources.yaml

```


* **네트워크 구동 실시간 확인:**
```bash
watch kubectl get pods -n calico-system

```


> 약 2분 후 `calico-kube-controllers` 및 `calico-node` 파드들의 STATUS가 `Running`으로 변하면 네트워크 구축이 완료된 것입니다.



---

## 6. 클러스터 상태 및 엔진 검증 (`kubectl get node -o wide`)

설치가 모두 완료되었는지 클러스터의 상태를 최종 조회합니다.

```bash
kubectl get node -o wide

```

* `myserver01`의 STATUS가 `Ready`로 표시되면 성공입니다.
* `CONTAINER-RUNTIME` 항목에 `containerd://1.6.24`가 명시되어 엔진이 올바르게 잡혔음을 확인할 수 있습니다.

## 💡 [NOTE] 마스터 노드에 파드(Pod) 배포 허용하기

기본 설계상 마스터 노드는 관리 전용 서버이므로 일반 컨테이너(파드) 배포를 금지하는 차단막(Taints)이 쳐져 있습니다. 테스트 목적으로 마스터 노드 혼자 파드를 띄우게 하려면 아래 명령어로 차단막을 해제(Untainted)할 수 있습니다.

* **Taints 상태 확인:**
```bash
kubectl describe node myserver01 | grep Taints
# 출력 결과: Taints: node-role.kubernetes.io/control-plane:NoSchedule

```


* **차단막 규칙 해제:**
```bash
kubectl taint nodes --all node-role.kubernetes.io/control-plane-

```


*(명령어 맨 뒤의 마이너스(`-`) 기호는 해당 규칙을 삭제한다는 의미입니다.)*

```
