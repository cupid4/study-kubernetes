# 9.7 잡과 크론잡

쿠버네틱스에서 잡(Job)과 크론잡(CronJob)은 앞서 살펴본 디플로이먼트(Deployment)와 완전히 반대되는 성격을 가진 오브젝트들입니다.

디플로이먼트는 24시간 365일 "절대 죽지 않고 켜져 있는 서버"를 만드는 것이 목적이라면, **잡과 크론잡**은 **"할 일 끝나면 쿨하게 종료되는 서버"를 만드는 것**이 목적입니다.

---

### 1. 잡 (Job)

* **한 줄 요약:** 정해진 태스크(일)를 **딱 한 번 또는 지정된 횟수만큼 실행하고 완료되면 스스로 종료**하는 관리자입니다.
* **왜 필요한가?** 계속 켜져 있을 필요 없이 데이터 백업, 이메일 대량 발송, 마이그레이션 같은 단발성 작업을 안전하게 끝내기 위해서입니다.

#### 🥟 쉬운 비유: "한정판 만두 100판 빚기 알바"

> 분식집 사장님이 단기 알바를 고용하며 **"오늘 딱 100판만 빚고 퇴근하세요"**라고 지시했습니다.
> 알바는 만두 100판을 다 빚으면 가게에 계속 남아있지 않고 바로 **퇴근(종료)**합니다. 만약 일을 하다가 중간에 다치거나 쓰러지면(파드 장애), 잡 관리자가 새 알바를 다시 데려와서 남은 만두 100판을 마저 채우도록 보장합니다.

---

### 2. 크론잡 (CronJob)

* **한 줄 요약:** 잡(Job)을 **시간표에 맞춰 주기적으로 실행**해 주는 스케줄러입니다. (Linux의 `cron` 기능을 쿠버네틱스로 가져온 것입니다.)
* **왜 필요한가?** 매일 밤 12시 정산 데이터 만들기, 매주 일요일 서버 청소하기처럼 반복적인 예약 작업을 자동화하기 위해서입니다.

#### 🏢 쉬운 비유: "매일 아침 문 여는 청소 대행업체"

> 사장님이 청소 업체와 계약을 맺었습니다. **"매일 아침 6시 정각에 와서 가게 청소를 끝내주세요"**
> 이 청소 업체가 바로 **크론잡**입니다. 아침 6시가 되면 청소부(잡/파드)를 가게로 보냅니다. 청소부는 청소가 끝나면 **퇴근(종료)**하고, 다음 날 아침 6시가 될 때까지 아무도 일하지 않습니다.

---

### 🔄 디플로이먼트 vs 잡 vs 크론잡 한눈에 비교

실무에서 성격에 따라 다르게 사용하는 핵심 차이점입니다.

| 구분 | 디플로이먼트 (Deployment) | 잡 (Job) | 크론잡 (CronJob) |
| --- | --- | --- | --- |
| **목적** | 24시간 켜져 있는 웹 서비스 | 단발성 작업 실행 및 완료 | 주기적인 예약 작업 실행 |
| **수명 (Life)** | 영원히 유지 (종료되면 에러로 간주) | 일이 끝나면 `Completed`로 종료 | 주기적으로 켜졌다가 일이 끝나면 종료 |
| **주요 예시** | Spring Boot 웹 서버, React 프론트 | DB 마이그레이션, 데이터 일회성 추출 | 매일 새벽 DB 백업, 매달 정산 이메일 발송 |
| **실행 기준** | 생성 즉시 실행 및 상시 유지 | 생성 즉시 1회성 실행 | `0 0 * * *` 같은 시간 조건 기준 실행 |

---

### ✍️ 크론잡의 시간 설정 방식 (쉬운 팁)

크론잡을 쓸 때는 `schedule: "0 0 * * *"` 처럼 별 5개를 이용한 시간표 설정을 사용합니다. 왼쪽부터 순서대로 [분] [시] [일] [월] [요일]을 뜻합니다.

* `*/5 * * * *` : 5분마다 실행
* `0 9 * * 1-5` : 월~금(평일) 아침 9시 정각마다 실행
* `0 0 1 * *` : 매달 1일 새벽 0시 정각마다 실행 (정산 작업 등)

쿠버네틱스는 이처럼 상시 켜져 있는 서버뿐만 아니라, 필요할 때만 켜서 컴퓨터 자원을 효율적으로 쓰고 꺼지는 '잡'과 '크론잡' 시스템을 제공하여 서버 비용을 극적으로 아낄 수 있게 도와줍니다.
